# 技术方案

## 默认技术路线

首版推荐使用 `React Native / Expo + TypeScript` 开发，一套代码同时支持 iOS 和 Android。

当前项目已按该方向初始化基础工程，包含 Expo 配置、TypeScript 配置、App 入口文件和基础主题文件。

选择原因：

- 适合跨平台移动 App。
- Expo 对图片选择、字体、资源管理、调试和发布流程更友好。
- TypeScript 可以减少后续维护错误。
- 对新手更容易逐步理解和推进。

## 前端重点

- 页面导航：文件夹列表、手账页面列表、手账编辑器、分享查看页。
- 手账编辑器：支持信纸背景、图片、文字、贴纸元素的添加、拖拽、缩放和保存。
- 素材管理：使用 JSON 索引统一管理贴纸、字体和背景，支持首屏常用素材、分页加载、分类缓存、图片预取、字体文件本地缓存和 `expo-font` 动态注册。贴纸库已拆分到 `src/features/materials/stickers.json`，当前包含 20 个分类、100 个示例贴纸；字体库已拆分到 `src/features/materials/fonts.json`，当前包含 20 款免费可商用字体。
- 主题系统：支持多套清新色系，主题只影响 App 外层界面，不破坏手账内容。
- 本地预览：编辑时必须即时看到页面效果。

## 后端方向

后端暂不立即实现，首版规划时保留以下能力：

- 用户账号：手机号或邮箱登录。
- 情侣绑定：邀请码生成、输入、绑定和解绑。
- 数据同步：手账页面、文件夹、素材引用和编辑数据同步到云端。
- 权限控制：情侣可编辑，第三方只读。
- 分享链接：生成可访问的只读页面。

后续可以评估使用 Supabase、Firebase 或自建后端。未确认前，不在代码中绑定不可替换的后端实现。

## 数据设计原则

- 手账页面不只保存成一张图片，也要保存可编辑结构。
- 图片、文字、贴纸都视为页面元素。
- 每个页面元素需要记录类型、位置、尺寸、旋转、层级和样式。
- 素材数据独立于页面元素保存，页面元素只引用素材或保存必要的展示数据，避免后续素材更新影响已完成页面。
- 文件夹负责组织页面，不直接承载复杂编辑逻辑。
- 分享链接只暴露只读数据。

## 稳定推进原则

- 先做单人本地编辑闭环，再做情侣同步。
- 先做基础拖拽排版，再做复杂排版。
- 先保存可编辑结构，再考虑导出图片。
- 每个阶段完成后必须验收，再进入下一阶段。
